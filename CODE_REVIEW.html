Part 1 — Steaming.Core & Steaming.Application

1.1 — Major Bugs
🔴 1.1.1 Silent wrong output — text[1..space] when space == -1 NOT A BUG ChatbotService.cs ~289

Re-examined: actual code at line 290 reads (space > 0 ? text[1..space] : text[1..]) — the ternary correctly handles space == -1 by falling to text[1..]. No fix required.

🟠 1.1.2 Fire-and-forget async delivery loses events on shutdown TwitchEventSubClient.cs ~28-34

PublishAsync() is called with ContinueWith(OnlyOnFaulted) but the task is never awaited or registered with a shutdown token. During application shutdown, in-flight deliveries are abandoned silently, dropping chat events.

🟠 1.1.3 TOCTOU race between WebSocket state check and ReceiveAsync ObsWebSocketService.cs ~84-102

ReadLoopAsync() checks _ws?.State == WebSocketState.Open and then calls ReceiveAsync(). The socket can transition to Closed between the two operations, producing an unhandled ObjectDisposedException.

🟡 1.1.4 Emoji rain color encoding has no bounds validation OverlayDispatcher.cs ~350-355

uint colorArgb is bit-shifted and cast to byte with no range validation. A malformed value produces wrong colors silently rather than an error.

1.2 — Orphaned / Dead Code
🟡 1.2.1 Dead method — TwitchReq() never called StreamDataService.cs ~126-132

TwitchReq(string url) is defined but HttpRequestMessage is constructed directly in PollOnceAsync() instead. Leftover from an earlier refactor.

🟡 1.2.2 Duplicate JSON helpers — RemoteKickBridgeClient vs KickAdapter RemoteKickBridgeClient.cs ~612 / KickAdapter.cs ~296

GetString() / GetBool() / GetInt() in RemoteKickBridgeClient duplicate TryGetString() and friends in KickAdapter. Neither references the other.

🟡 1.2.3 Unused Platform assignment in RemoteKickBridgeClient RemoteKickBridgeClient.cs ~318

Platform is extracted from the JSON root and logged, but never used for any branching logic. Confusing dead assignment.

1.3 — Memory Leaks
🟠 1.3.1 Static singleton event never unsubscribed (OverlayDispatcher) FIXED v0.9.9 OverlayDispatcher.cs ~67

Stop() method added to OverlayDispatcher. Calls _bus.Unsubscribe(OnEvent) and EmoteCache.Instance.OnDownloadComplete -= OnEmoteDownloadComplete. Resets _started so Start/Stop cycles are safe.

🟠 1.3.2 EventBus subscriptions accumulate across Start/Stop cycles FIXED v0.9.9 SoundDispatcher.cs / ViewerListService.cs

SoundDispatcher: Stop() added — calls _bus.Unsubscribe(OnEvent). ViewerListService: Subscribe moved from constructor to Start(); Stop() calls _bus.Unsubscribe(OnEventAsync) before stopping the timer, so repeated Start/Stop cycles leave no stale handlers.

🟡 1.3.3 ViewerListService timer handler not removed before Stop() ViewerListService.cs ~50-54

_timer.Elapsed += ... is registered in Start(). Stop() disposes the timer but does not detach the handler first. On a subsequent Start(), a new handler is added to a new timer, and the old one may fire during the dispose window.

1.4 — Performance
🟡 1.4.1 Uncompiled Regex in hot path + double count() ChatbotService.cs ~410-430

Regex.IsMatch() is called per-message without a pre-compiled static instance — regex compilation on every message adds significant latency. text.Count(char.IsUpper) appears to be called twice on the same string in the same method. Both are GC-pressure sources in a high-frequency path.

Fix: Declare private static readonly Regex _pattern = new(..., RegexOptions.Compiled); and cache the uppercase count.

🟡 1.4.2 Twitch follower API polled unconditionally every 30 s StreamDataService.cs ~181-225

The follower endpoint is called every poll cycle (~2880 calls/day) regardless of whether the newest follower has changed. A comparison against _lastTwitchNewestFollower before making the call would skip ~99% of these requests.

🟡 1.4.3 All 11 label objects rebuilt on every SendAllLabels tick OverlayDispatcher.cs ~243-259

SendAllLabelsAsync() reconstructs and re-sorts all label values on every call even when only one label changed. Delta tracking would allow sending only changed labels.

1.5 — Thread Safety
🟠 1.5.1 Timer tick race can send duplicate messages ChatbotService.cs ~108

_timerTick.Elapsed fires CheckTimers() without locking. If a slow handler causes two ticks to overlap, both can pass the cooldown check simultaneously and send the same timer message twice.

🟡 1.5.2 ViewModelBase.Set() has no thread guard ViewModelBase.cs ~10-14

Set() reads, compares, and writes a backing field with no synchronisation. Any ViewModel property written from a background thread while the UI thread reads it is an unguarded race. WinUI's dispatcher normally prevents this but is not enforced here.

1.6 — Easy Wins
🟢 1.6.1 Authorization header construction duplicated in 3+ services ModerationService / TwitchAdapter / StreamManagementService

All three build Authorization headers independently. Extract to a TwitchAuthorizationHelper.BuildHeaders() static method.

🟢 1.6.2 ?? "" null-coalescing repeated ~10 times OverlayDispatcher.cs ~248-256

StreamData?.RecentFollower ?? "" and similar patterns appear ~10 times in one method. Assign to local variables at the top.

🟢 1.6.3 Bare catch blocks silently swallow exceptions TwitchBadgeService / DebugLogFile / ObsWebSocketService

Multiple catch { } blocks discard all exceptions with no logging. At minimum: catch (Exception ex) { DebugLog(ex.Message); }

🟢 1.6.4 Manual JsonDocument traversal — candidate for source generators TwitchEventSubClient.cs / KickAdapter.cs

Heavy JsonDocument.GetProperty() chaining. These are prime candidates for [JsonSerializable] source generators (C# 11+), eliminating allocations and improving parse speed.

Part 2 — Steaming.WinUI

2.1 — Major Bugs
🟠 2.1.1 Background task uses disposed repository (AnalyticsPage) AnalyticsPage.xaml.cs ~141, 259-265

A background task captures _repo at line 260. OnNavigatedFrom sets _repo = null after disposing it (line 52-53). If the page is navigated away while the task is running, the DispatcherQueue.TryEnqueue callback executes repo.GetSnapshots() on the already-disposed repository. The null check at line 265 is after the call, not before it.

Fix: Capture var repo = _repo; before the background task, null-check it immediately, and cancel the task in OnNavigatedFrom.

2.2 — Orphaned Code
🟡 2.2.1 Empty SelectionChanged handlers wired in XAML OverlaysPage.xaml.cs ~336 / GoalsLabelsPage.xaml.cs ~28

LabelList_SelectionChanged() is wired to SelectionChanged in XAML on both pages but has a completely empty body. The binding adds dispatch overhead with zero benefit. Remove the handler and the XAML attribute.

2.3 — MVVM Violations
🟡 2.3.1 Goal CRUD logic embedded in BuildGoalExpander() code-behind OverlaysPage.xaml.cs ~216-294

Inline click handlers inside BuildGoalExpander() perform save, push-to-overlay, and delete operations directly. This is business logic that belongs in the ViewModel as commands.

🟡 2.3.2 Chat preview rendering logic lives in ChatPage code-behind ChatPage.xaml.cs ~141-232

RenderPreview() performs color parsing, font construction, and control creation from configuration state. Should be a ViewModel helper or shared service.

2.4 — Memory / Resource Issues
🟠 2.4.1 GIF frame cache has no size limit (potential OOM) OverlaysPage.xaml.cs ~860-873

BitmapToWriteable() converts GIF frames to WriteableBitmap and stores them in an unbounded cache. A 1920×1080 GIF at 60 frames consumes ~497 MB. Multiple such GIFs in an overlay set can exhaust working memory silently. Implement an LRU cache with a configurable byte-size cap.

2.5 — Performance
🟠 2.5.1 Per-frame RotateTransform allocation in animation loop OverlaysPage.xaml.cs ~694 / AlertEditorPage.xaml.cs ~403

A new RotateTransform object is allocated every animation frame for every element, even when the angle has not changed. This is the highest-frequency GC allocation source in the UI layer. Transforms should be created once and mutated (.Angle = newValue).

🟡 2.5.2 Synchronous GIF frame loading blocks UI thread OverlaysPage.xaml.cs ~824-843

LoadGifFrames() performs synchronous file I/O. A GIF with 50+ large frames will stall the UI thread visibly during load. Should be async with await.

🟡 2.5.3 ScrollIntoView triggers full layout pass on every chat message DashboardPage.xaml.cs ~69

ChatList.ScrollIntoView() forces a layout pass per message. On a busy chat (100+ msg/min) this causes continuous layout recalculation. Debounce to 100–200 ms intervals, or only scroll when the list is already at the bottom.

2.6 — Duplicated Code (Easy Wins)
🟢 2.6.1 Status bar refresh logic duplicated in MainWindow and DashboardPage MainWindow.xaml.cs ~58-74 / DashboardPage.xaml.cs ~144-160

RefreshStatusBar() and RefreshPlatformDots() are implemented identically in two files. Extract to a shared static helper.

🟢 2.6.2 Color parsing duplicated across 3+ pages OverlaysPage / AlertEditorPage / ChatPage

TryParseColor() and ParseBrush() are copy-pasted between pages. Move to a WinUIColorHelpers static class.

🟢 2.6.3 GIF loading utilities duplicated across OverlaysPage and AlertEditorPage OverlaysPage.xaml.cs ~824-873 / AlertEditorPage.xaml.cs ~457-506

LoadGifFrames(), GetGifDelays(), and BitmapToWriteable() are copy-pasted. Extract to a GifFrameLoader utility class — this also means the LRU fix (2.4.1) only needs to be implemented once.

🟢 2.6.4 BuildTextGrid / text span rendering duplicated OverlaysPage.xaml.cs / AlertEditorPage.xaml.cs

Shadow/outline text grid construction is nearly identical in both files. Extract to a TextGridBuilder helper — any future span rendering changes only need to be made once.

Part 3 — Steaming.Data

3.1 — SQL / Data Issues
🟡 3.1.1 DDL migration uses string interpolation — SQL injection landmine AnalyticsRepository.cs ~101-120

ALTER TABLE ... ADD COLUMN {col} {def} uses string interpolation. The values are currently hardcoded tuples (safe), but the pattern is a landmine if column names are ever sourced from user input or a config file. Add a comment explicitly documenting that these values must remain compile-time constants, or switch to a whitelist validation.

🟡 3.1.2 Unused platform parameter implies missing per-platform tracking AnalyticsRepository.cs ~257-278

IncrementFollows(long sessionId, string platform) and IncrementSubs() accept a platform parameter but the SQL ignores it and increments the combined column. The API implies per-platform tracking that does not exist. Either implement it (per CLAUDE.md Rule 17 — both platforms must be tracked separately) or remove the parameter.

3.2 — Performance
🟡 3.2.1 3 database round-trips in GetAllTimeStats() instead of 1 AnalyticsRepository.cs ~384-407

GetAllTimeStats() makes three separate queries: main stats + GetPlatformFollows() + GetPlatformSubs(). A single SELECT platform, SUM(total_follows), SUM(total_subs) FROM stream_sessions GROUP BY platform returns all data in one round-trip.

🟡 3.2.2 IncrementFollows / IncrementSubs lack transaction protection AnalyticsRepository.cs ~257-278

AddSnapshot() correctly uses BeginTransaction(), but these two increment methods do not. Under a sub bomb or rapid follow events, concurrent calls can interleave inconsistently. Wrap in a transaction or consolidate into the snapshot write.

Part 4 — OBS Plugin (C++)

4.1 — Critical Correctness
🔴 4.1.1 Off-by-one buffer overread in emoji_rain_trigger emoji_rain_source.cpp ~330

When payload.size() == 7 + slen, the bounds check passes but the subsequent array access reads one byte past the end of the vector — undefined behavior, potential crash or information disclosure.

// CURRENT (wrong):
if ((size_t)(7 + slen) > payload.size()) return;
uint8_t count = payload[7 + slen];  // OOB when 7+slen == payload.size()

// CORRECT:
if ((size_t)(7 + slen) >= payload.size()) return;
4.2 — Performance / Memory
🟠 4.2.1 GPU texture destroyed and recreated every render frame alert_source.cpp ~313-322

The render function calls gs_texture_destroy(d->texture) then gs_texture_create(...) every frame unconditionally. This causes GPU memory thrashing. The texture should be cached and only recreated when the frame dimensions change.

🟡 4.2.2 Per-frame std::vector heap allocation in render loop chat_source.cpp ~643-656

std::vector<uint8_t> pixels is stack-declared (heap-allocated via the vector) fresh every frame. The correct pattern is demonstrated elsewhere in the codebase (alert_source.cpp uses a member mixBuffer that is resized grow-only and reused). Apply the same pattern here.
